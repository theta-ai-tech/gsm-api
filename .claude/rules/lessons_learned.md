# Lessons Learned

- When one service action can emit multiple ticker event types, scope test assertions to the event type under test instead of asserting on total `ticker_repo.add` calls or total `ticker` documents.
- When writing manual testing instructions in PR descriptions that reference seeded match IDs, always use the actual IDs from `tools/seed_data.py`. The only seeded match in `PENDING_CONFIRMATION` status is `match_pending` (underscore). Do not invent IDs like `match_001`.
- Before merging a PR, ensure the feature branch is rebased onto main if any other PRs landed after the branch was created. Stale branches carry unformatted files from older main states, causing CI lint failures even when the PR diff itself is clean. In the developer cron, run `git -C {WORKTREE_PATH} merge origin/main` before pushing the final commit when another PR has been merged since this branch was cut.
- `make test` works from the repo root as-is — do NOT modify `pytest.ini` or add `pythonpath` settings. The editable install (`pip install -e api[dev]`) already puts `app` on the import path. If tests fail with import errors, the fix is `make install` (reinstall the editable package), not patching pytest config.
- When running tests for a feature branch in a git worktree, `make -C {MAIN_REPO_ROOT} test` does NOT exercise the branch's code. The main checkout stays on `main`, the editable install resolves `app` from `{MAIN_REPO_ROOT}/api/app`, and pytest collects from `{MAIN_REPO_ROOT}/tests/` — none of which see the worktree's changes. To actually test the branch, run from the worktree with the main venv and a `PYTHONPATH` override: `PYTHONPATH={WORKTREE_PATH}/api make -C {WORKTREE_PATH} VENV={MAIN_REPO_ROOT}/.venv test`. Verify the worktree was loaded by importing `app` once and checking `app.__file__` starts with `{WORKTREE_PATH}/api/app`.
- When the autopilot skill says to invoke `/next-issue`, delegate to the `gsm-backend-developer` agent — do not implement inline. The CLAUDE.md delegation table is explicit: new features go to the agent.
- When adding a field to a response model, grep for EVERY construction site of that model — not just the Firestore mapper. Router files keep local field-by-field converters (e.g. `_league_to_browse_card` in `routers/leagues.py`) that duplicate `repos/mappers.py`; a field added only to the mapper silently falls back to the model default on the router path. Shape tests that assert key sets don't catch this — assert the field's VALUE from a non-default fixture.
- For account-deletion / multi-step destructive endpoints, ordering is a correctness concern: do all data erasure (Firestore subcollections + tombstone) FIRST while the caller's ID token is still valid, and destroy the identity LAST. A mid-flow failure then leaves the token valid and the request idempotently retryable, never a locked-out account with orphaned PII. Also make identity destruction a SINGLE op: `delete_user` alone locks the caller out (prod verifies with `verify_id_token(check_revoked=True)`, which 404s a deleted user) and drops refresh tokens — a separate `revoke_refresh_tokens` before it is redundant and adds a revoked-but-not-deleted window (revoke succeeds, delete fails → signed out but PII remains). Don't pair revoke+delete.
